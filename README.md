# Lightweight Motion-QA: LLM-Guided Question Answering over 3D Human Motion

> **CPU-only prototype** that answers natural-language questions about 3D human motion using simple PyTorch motion analysis + an LLM planner and explainer.  
> Motion data comes from a small subset of the **BABEL** dataset.

---

## 1. Overview

Modern large-language-model (LLM) systems can describe images, videos, and code—but **understanding human movement** is still a relatively underexplored space.

This project implements a **lightweight Motion Question Answering (Motion-QA)** system:

- Input:
  - a 3D skeleton motion sequence (e.g., a short clip from BABEL, represented as `T × J × 3` joint positions),
  - a natural-language question like:
    - “How many times does the person sit down?”
    - “Does the person move more forward or sideways?”
    - “Which limb moves the most in this clip?”
- Output:
  - a concise, human-readable answer generated by an LLM.

Instead of training a huge transformer or diffusion model, we:

1. **Extract interpretable motion features** in pure PyTorch on CPU (displacements, velocities, joint activity, simple events).
2. Use an **LLM as a planner** to decide *which* motion-analysis module (tool) to use for a given question.
3. Run a deterministic **motion QA module** (e.g., `count_sit_events`, `dominant_direction`, `most_active_limb`) to get a raw answer.
4. Use the **LLM as an explainer** to convert that raw answer into a natural-language response.

Conceptually, this is inspired by recent work on **Human Motion Question Answering** (HumanMotionQA / BABEL-QA), but re-imagined as a **GPU-free, interpretable, and modular** system using the BABEL dataset for motion.  

---

## 2. Key Features

- **LLM-guided reasoning over motion**  
  The LLM doesn’t see raw coordinates; instead it picks which motion tool to call and then verbalizes the results.

- **3D human motion-aware**  
  Works on skeleton-based motion sequences (`T × J × 3`), sourced from the **BABEL** dataset.

- **Modular QA “tools”**  
  Questions are answered by explicit, interpretable modules:
  - `dominant_direction` – forward/backward/left/right?
  - `most_active_limb` – which limb moves most?
  - `count_sit_events` – how many sit/squat-like events?

- **CPU-only & lightweight**  
  No GPUs required. All motion processing is simple tensor math; the heavy lifting is done remotely by the LLM provider.

- **Extensible**  
  You can easily add:
  - new motion features,
  - new QA modules,
  - new question templates and synthetic QA pairs.

## 4. Method

### 4.1. Motion Representation

Each motion clip is represented as a tensor:

```text
S ∈ ℝ^{T × J × 3}
```

- `T`: number of timesteps
    
- `J`: number of joints
    
- `3`: (x, y, z) coordinates or root-centered coordinates in meters
    

Raw BABEL motion (e.g., SMPL/AMASS) is preprocessed into this skeleton format and stored as `.npy` files.

### 4.2. Feature Extraction

`motion_qa/features.py` implements CPU-only functions such as:

- Root joint trajectory & displacement (`final_root - initial_root`)
    
- Per-joint **path length** (how much each joint moved)
    
- Per-joint **mean/max speed**
    
- Simple **event detectors** (e.g., hips dropping below a threshold → sit/squat events)
    

Example (conceptual):

```python
features = {
    "duration_sec": ...,
    "root_displacement": ...,
    "per_joint_path_length": ...,
    "per_joint_speed_mean": ...,
    "per_joint_speed_max": ...,
    "sit_event_count": ...,
}
```

These features provide a **semantic summary** that the QA modules can reason about.

### 4.3. Motion QA Modules

`motion_qa/modules.py` defines small, deterministic “tools” that answer specific query types:

- `dominant_direction(motion, features, params={})`
    
    - Returns which axis dominates the movement: `{"value": "forward", "details": {...}}`.
        
- `most_active_limb(motion, features, params={})`
    
    - Aggregates per-joint motion by limb (left/right arm/leg) and returns the most active one.
        
- `count_sit_events(motion, features, params={})`
    
    - Counts hip height dips over time to approximate sit/squat events.
        

Each module returns a **structured raw answer** (dict) rather than plain text so the LLM can format it cleanly later.

### 4.4. LLM Planner

`motion_qa/planner.py` is responsible for:

- Taking a **user question** (e.g., “Which limb moves the most?”),
    
- Knowing what tools are available,
    
- Asking the LLM to output a small JSON “plan”:
    

```json
{"tool": "most_active_limb", "params": {}}
```

The prompt given to the LLM:

- Describes the available tools and their purpose,
    
- Asks for **only** a JSON object (no additional text),
    
- Includes a few example questions → tool mappings.
    

### 4.5. LLM Answerer

`motion_qa/answerer.py`:

- Takes the original question and the raw tool output,
    
- Asks the LLM to produce a **1–2 sentence explanation**:
    

> Question: “Does the person move more forward or sideways?”  
> Raw: `{"value": "forward", "details": {"lr": 0.3, "fb": 2.1}}`  
> Answer: “The person moves mostly forward, with only a small sideways displacement.”

This way, all heavy language work happens inside the LLM; your code does the numeric reasoning.

## 5. Using BABEL

This project assumes you have access to a small subset of **BABEL: Bodies, Action and Behavior with English Labels**, a motion capture dataset with **~43 hours of AMASS mocap sequences annotated with English action labels at both sequence and frame level**.

 **Data access & license**  
> You must obtain BABEL from the official project website and agree to their license terms. This repository does **not** redistribute BABEL data.

Helpful resources (see BABEL site/paper for full details):

- BABEL project site:  
    `https://babel.is.tue.mpg.de/`
    
- BABEL paper (CVPR 2021):  
    “BABEL: Bodies, Action and Behavior with English Labels” – Punnakkal et al.
    

### 5.1. Preparing a Small BABEL Subset

You don’t need all of BABEL for this project. A simple workflow:

1. **Download** BABEL data according to the instructions on the official website.
    
2. Pick a **modest subset** of motion sequences (e.g., 50–200 clips) to keep things CPU-friendly.
    
3. Convert each sequence into a skeleton format (e.g., `T × J × 3`) and save as `.npy` in `data/babel_subset/motions/`.
    
4. Create `data/babel_subset/metadata.json` that looks like:
    

```json
[
  {
    "id": "clip_0001",
    "motion_file": "clip_0001.npy",
    "questions": [
      {
        "q": "How many times does the person sit down?",
        "a": "2",
        "type": "count_sit_events"
      },
      {
        "q": "Does the person move more forward or sideways?",
        "a": "forward",
        "type": "dominant_direction"
      }
    ]
  },
  ...
]
```

You can create these QA pairs:

- manually,
    
- via heuristics using BABEL action labels (e.g., count “sit” actions),
    
- or semi-automatically using an LLM (e.g., from motion stats / labels).

## 6. Installation

### 6.1. Clone the Repository

```bash
git clone https://github.com/Sanjana2070/Lightweight-Motion-QA.git
cd Lightweight-Motion-QA
```

### 6.2. Create a Virtual Environment (optional but recommended)

```bash
python -m venv .venv
source .venv/bin/activate   # on Linux / macOS
# .venv\Scripts\activate    # on Windows
```

```bash
pip install -r requirements.txt
```

A minimal `requirements.txt` might look like:

```text
torch              # CPU build
numpy
tqdm

```

## 7. Configuration (LLM API Keys)

Set your LLM provider API key as an environment variable, for example:

```bash
export OPENAI_API_KEY="sk-..."   # or any provider-specific key
```

The `planner.py` and `answerer.py` modules should read the key from environment variables (or `.env` via `python-dotenv`).

## 8. How to Run

### 8.1. Prepare Data

Make sure you have:

- `data/babel_subset/motions/*.npy`
    
- `data/babel_subset/metadata.json`
    

You can write & run your own preprocessing script (or use `scripts/preprocess_babel_subset.py` once implemented).

### 8.2. Test the Pipeline with `run_demo.py`

Once data and API key are ready:

```bash
python -m scripts.cli_app
```
    

**Example expected console output:**

```text
Loaded motion clip: clip_0001  (T=240, J=22)
User question: Which limb moves the most in this clip?

[Planner] Selected tool: most_active_limb
[Module] Raw answer: {'type': 'categorical', 'value': 'right_leg', 'details': {...}}

Final answer:
"The right leg is the most active limb in this motion, moving more than the other arms and legs over the duration of the clip."
```

## 9. Roadmap / Possible Extensions

- **More QA modules**  
    Add modules that handle:
    
    - speed/tempo (“Is this motion fast or slow?”),
        
    - symmetry (“Is left/right usage balanced?”),
        
    - repetition (“How many jumping jacks?”).
        
- **Tiny learned models (still CPU-friendly)**
    
    - Train small classifiers on top of features (e.g., “walk vs run vs sit”).
        
    - Use them as additional tools the LLM can call.
        
- **Better dataset integration**
    
    - Use BABEL’s action labels more systematically to generate Q/A pairs.
        
    - Explore compatibility with BABEL-QA style questions.
        
- **Improved visualization**
    
    - Animate skeletons over time using `matplotlib` or `plotly`.
        
    - Optionally render gif/mp4 for each answered question.
        
- **Beyond BABEL**
    
    - Extend to other mocap datasets or to real-time pose estimates from video.
